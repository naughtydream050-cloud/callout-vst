# Self-Evaluation Harness — OpenHarness型自己採点基準
> **OpenHarness Evaluation Standard v1.0**
> Gemma 2がUI・DSP・ロジック実装を「10点満点」で採点するための厳格なチェックリスト。
> **9点未満はOrchestratorへの受理を拒否し、自動リトライを要求する。**

---

## OpenHarness 思想

```
従来の開発レビュー: 「だいたいできてればOK」「動けばいい」
OpenHarness:       定量的なスコアで合否を判定。感覚論を排除。
                   9/10 以上でなければ次のステージには進まない。
                   理由: 8点の設計が10回実装を繰り返すより、
                          10点の設計が1回で通る方がトークン効率が高い。
```

---

## 採点方式

- **満点**: 10点
- **合格ライン**: 9点以上（= 即時 PASS）
- **条件付き合格**: 8点（軽微な懸念のみ。Orchestratorの判断で通過可）
- **不合格**: 7点以下（= REJECT。修正・再提出が必須）

各カテゴリは独立して採点し、最後に加重平均を計算する。

## カテゴリA — UI設計評価（blueprint.md 評価用）

**配点: 4点**

### A1. 座標精度 (1点)
| 条件 | 点数 |
|------|------|
| 全コンポーネントがキャンバス内に収まる | +0.4 |
| ノブがW=H（正方形）になっている | +0.3 |
| X, Y ≥ 0（負の座標なし） | +0.3 |

### A2. パラメータID整合性 (1点)
| 条件 | 点数 |
|------|------|
| 重複するパラメータIDがない | +0.5 |
| IDが英数字・アンダースコアのみで構成 | +0.3 |
| デフォルト値が min〜max 範囲内 | +0.2 |

### A3. DSPシグナルチェーン一貫性 (1点)
| 条件 | 点数 |
|------|------|
| Input → [処理] → Output の流れが明記されている | +0.5 |
| 各処理ブロックが blueprint に対応している | +0.5 |

### A4. ノブ仕様完全性 (1点)
| 条件 | 点数 |
|------|------|
| 全ノブに画像ファイル名が指定されている | +0.4 |
| 回転範囲（startAngle/endAngle）が標準値か記載がある | +0.3 |
| アーク色・グロー仕様が定義されている | +0.3 |

## カテゴリB — コード品質評価（実装コードのレビュー用）

**配点: 4点**

### B1. LookAndFeel 実装品質 (1点)
| 条件 | 点数 |
|------|------|
| KnobLookAndFeel が独立ファイルに分離されている | +0.3 |
| `ScopedSaveState` で変換を適切に分離している | +0.3 |
| デストラクタで `setLookAndFeel(nullptr)` を呼んでいる | +0.4 |

### B2. リソース管理 (1点)
| 条件 | 点数 |
|------|------|
| 画像ロードがコンストラクタで1回のみ | +0.4 |
| `knobImage.isValid()` でフォールバックを用意している | +0.3 |
| メモリ解放忘れ・ダングリングポインタのリスクがない | +0.3 |

### B3. CMakeLists.txt 正確性 (1点)
| 条件 | 点数 |
|------|------|
| 新規 .cpp が `target_sources` に登録されている | +0.4 |
| `/utf-8` フラグが MSVC 向けに設定されている | +0.3 |
| プロジェクトパスが ASCII パスであることを前提としている | +0.3 |

### B4. エラーハンドリング (1点)
| 条件 | 点数 |
|------|------|
| 境界値（スライダーの min/max）での動作が考慮されている | +0.4 |
| BinaryData 取得失敗時のフォールバックが実装されている | +0.3 |
| 例外・null チェックが適切に行われている | +0.3 |

## REJECT 時の自動リトライフロー

```
Gemmaが [REJECT] を返した
         ↓
改善必須事項リストをOrchestratorに報告
         ↓
Orchestratorが blueprint.md / コードを修正
         ↓
gemma_bridge.py --mode eval を再実行
         ↓
[PASS] になるまでループ（最大3回）
         ↓
3回でも [REJECT] → 人間にエスカレーション
```
